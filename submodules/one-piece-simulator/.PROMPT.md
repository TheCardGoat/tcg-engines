# One Piece Card Coverage Gate

You are running a long-window execution task for the One Piece TCG simulator.

Your job is to validate card behavior by writing functional card tests, not by
only reading card text.

Use `.agents/skills/op-rules` as the source of truth for One Piece TCG rules
questions.

## Mandatory Test Harness

All new or updated card behavior tests must use the existing engine fixture harness:

`packages/engine/src/testing/test-engine.ts`

Use `OnePieceTestEngine` and its fixture helpers to simulate real user
interactions through the engine, such as:

- creating a fixture state with cards in hand, deck, life, trash, character
  area, stage, DON, etc.
- playing a card with `engine.playCard(...)`
- attaching DON with `engine.attachDon(...)`
- declaring attacks with `engine.declareAttack(...)`
- activating effects with `engine.activateEffect(...)`
- inspecting results with `engine.getState()` or `engine.getView(...)`

Do not satisfy this task with tests that only inspect card metadata, card text,
ability definitions, snapshots, or static exports.

A card is only covered when a test proves its implemented behavior works through
the engine harness.
For each card we should have a test file under packages/engine/src/cards that matches the same fodler structure of the cards

## Required Test Location And Shape

For every card under `packages/cards/src/cards`, write or update a corresponding
card behavior test under:

`packages/engine/src/cards`

Each card must have at least one test case in that engine card-test area. The
test case must execute the card's ability through `OnePieceTestEngine`.

Do not mark a card as complete because its card definition exists, because its
text was reviewed, or because its ability object appears to match the text. The
test must drive the ability through engine commands and assert the resulting game
state.

For cards with triggered abilities, the test must create the trigger condition
and execute the trigger. For cards with activated abilities, the test must
activate the ability. For cards with static or continuous effects, the test must
put the card into the relevant zone and assert the continuous behavior is active.

If an ability creates a temporary effect that lasts "during this turn", the test
must also advance or end the turn and assert the temporary effect expires.

Example requirement:

- For `packages/cards/src/cards/EB01/characters/006-tony-tony-chopper.ts`, the
  card text says: `[DON!! x2][When Attacking] Give up to 1 of your opponent's
Characters -3000 power during this turn.`
- The test must build a fixture with Tony Tony Chopper in play, at least two DON
  attached to Chopper, and a valid opponent character target.
- The test must declare an attack with Chopper through `engine.declareAttack(...)`
  or the supported engine interaction for attack triggers.
- The test must resolve the ability against the opponent character and assert
  that character has `-3000` power during the turn.
- The test must pass or end the turn and assert the opponent character's power
  returns to normal.

## Source Of Truth

For each card, compare:

1. the printed card text
2. the implemented ability definition
3. the actual behavior when driven through `OnePieceTestEngine`

If the ability definition does not match the printed card text, fix the
implementation.

If the implementation looks correct but does not work when executed through the
test harness, fix the engine/card integration or the ability implementation.

Treat printed card text as the source of truth for card behavior.

## Work One Card At A Time

For each card:

1. Read the card file.
2. Read the printed card text.
3. Read the ability implementation.
4. Identify the normal happy-path user flow.
5. Write or update a unit test using `OnePieceTestEngine`.
6. The test must execute the card behavior through engine commands.
7. Assert the observable game-state result.
8. Run the relevant test file.
9. Update `CARD_TEST_COVERAGE_TODO.md` with what was done and why.

Do not batch many cards as "reviewed" without writing or confirming
harness-based functional tests.

## What Counts As A Valid Card Test

A valid card test must prove behavior by exercising the engine.

Good examples:

- A character with an On Play effect is placed in hand, played with
  `engine.playCard(...)`, and the resulting state is asserted.
- An Activate Main effect is placed on board, activated with
  `engine.activateEffect(...)`, and the resulting state is asserted.
- An attack-triggered effect declares an attack with
  `engine.declareAttack(...)`, resolves the effect, and asserts the result.
- A card requiring DON uses fixture DON plus `engine.attachDon(...)` or relevant
  cost payment commands.
- A targeting effect sets up valid targets, performs the user-like target
  selection flow supported by the engine, and asserts the chosen target changed
  correctly.

Invalid examples:

- A test that only imports the card and checks its text.
- A test that only checks the ability object exists.
- A test that snapshots card metadata.
- A test that bypasses the engine and directly calls low-level effect internals,
  unless it is a module-level engine test rather than a card test.
- A test that says the card is covered because the implementation visually
  matches the text.

## Card Test Scope

Card-level tests should cover the happy path only.

Happy path means the normal successful user interaction:

- card is available in the right zone
- costs can be paid
- valid targets exist
- the action is selected or activated
- the expected result happens

Do not add broad edge-case testing at the card level.

Edge cases belong in focused module tests for:

- targeting
- condition evaluation
- triggers
- costs
- replacement effects
- timing validation
- rule enforcement

If a card exposes a missing edge case in a shared module, add or fix the
appropriate module test separately.
