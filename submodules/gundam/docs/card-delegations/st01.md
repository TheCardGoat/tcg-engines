## Blocker buckets

### A. Command-card 【Burst】 dispatch — not yet wired

Command cards' 【Burst】 clauses (e.g. `activateTiming: main`) are not
re-activated when a shield is destroyed. The engine's shieldDestroyed
enqueue path fires for the destroyed card's own trigger set, but command
cards don't round-trip through the play-from-hand pipeline on burst.

Representative cards:

- `command/014-unforeseen-incident.ts` (Burst → Activate Main)

### B. `<Blocker>` direct-attack gating

`<Blocker>` is declared in card data but no card-level test redirects a
direct attack onto a blocker. The engine's battle-phase suite explicitly
tags this as a TODO.

Representative cards:

- ~~`unit/008-demi-trainer.ts`~~ ✅ Converted (Wave 10) — <Blocker> re-targets attack (defender-vs-attacker pattern via `declareBlock`); Wave-15 adds `expectAttackRedirectedTo` assertion.
- `unit/009-zowort.ts` — granted-keyword variant still blocked (constant `grantKeyword` self-target not landed in continuous-effects registry at battlefield entry).

### C. Pilot-resident `attack` triggers

Pilot triggers with `timing: ["attack"]` fire when the paired unit declares
an attack. No card-level test drives pilot-attack-trigger effects
end-to-end (needs assignPilot + enterBattle + stat/resource assertions in
one flow).

Representative cards:

- `pilot/011-suletta-mercury.ts` (setActive on a friendly Resource)
