## When to use this skill

- Periodically (weekly / before a release).
- After a significant refactor that touched conventions or package layout.
- When `pnpm run check:harness` passes but you suspect docs are still wrong.

Don't run it for every PR — the agent's job is to detect **semantic** drift, which is expensive to evaluate. Computational drift (broken links, stale paths) is caught cheaply by `tools/harness/check-doc-drift.mjs`; run that first.

## Inputs

- `SCOPE` — which corpus to audit:
  - `docs` (default) — `docs/**/*.md`, `AGENTS.md`, `README.md`.
  - `skills` — `.claude/skills/*.md`.
  - `all` — both.
- `SINCE` — git revision range to ground the audit in. Default: last 30 commits. Use this to focus on areas the code recently changed.

## Procedure

1. **Establish a baseline** — read [`docs/architecture.md`](../../docs/architecture.md) and [`docs/design-docs/core-beliefs.md`](../../docs/design-docs/core-beliefs.md). These are the most-likely-load-bearing docs; if they're stale, everything downstream is suspect.
2. **Run cheap checks first** — `pnpm check:harness` and `pnpm check:doc-drift`. Don't waste tokens on what these already find.
3. **Survey recent code changes** — `git log --stat --since="30 days ago"` (or `${SINCE}`). Identify packages/areas with significant churn. Those are the most likely places docs are stale.
4. **For each doc in SCOPE**, ask:
   - Does it describe a pattern that the recent changes have invalidated?
   - Does it mention a symbol/function that has been renamed or removed? (Grep for the symbol; if zero hits, the doc is stale.)
   - Does it state an invariant the code no longer upholds? (Spot-check: pick the invariant, find a representative file, read it.)
   - Are example snippets still correct? Type-check them mentally against current types.
5. **Output a punch list** — for each issue, one of: (a) trivial fix (PR-able), (b) needs human input, (c) doc should be deleted. Be specific about which lines.

## What "stale" looks like in this repo

Concrete patterns to watch for:

- **`packages/engine/src/index.ts` curated surface**: docs that name an export which has been removed or renamed.
- **`MatchRuntime` API surface**: method names change occasionally; docs sometimes lag.
- **Card-delegation docs**: `docs/card-delegations/*.md` were point-in-time summaries; many cards they list as TODO are now implemented. Spot-check by grepping the card path.
- **Skill files**: `.claude/skills/implement-card.md` and friends reference test helper functions (`createMockUnit`, `seedShieldsFromDeck`, etc.). When those helpers move, the skill needs updating.
- **`CLAUDE.md`**: auto-generated by `vp config` but agents sometimes add custom rules to it that get clobbered. Check for hand-written sections that disappeared.

## Output format

```
## Doc-gardening verdict: CLEAN | DRIFTED

### Cheap checks
- check:harness: ✓/✗
- check:doc-drift: ✓/✗

### Inferential findings
1. `<doc>:<line>` — <what's stale> — <suggested fix>
2. …

### Should be deleted
- `<doc>` — <reason>

### Needs human input
- `<doc>` — <ambiguity>
```

## When you should fix vs. flag

- **Fix in place**: typos, broken link targets, paths that moved, references to renamed symbols.
- **Open a PR with the proposed change**: anything that changes the meaning of a doc (you might be wrong about what the code does).
- **Flag for human**: deletions of design docs, changes to `core-beliefs.md`, anything in [`docs/exec-plans/completed/`](../../docs/exec-plans/completed/) (those are historical, not living).

Don't delete a doc you find confusing — find out why it was written first. Likely it's tracking something a previous agent or human got bitten by.
